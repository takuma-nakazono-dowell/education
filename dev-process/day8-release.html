Day 8: リリースと保守

← 前へ: テスト ← トップに戻る

リリース（デプロイ）とは

開発・テストが完了したシステムを、実際にユーザーが使える本番環境に展開すること。

リリース前の確認事項

  • リリース判定会議: 全テストが完了し、残バグが基準以内か確認する
  • リリース計画書: いつ・誰が・どの順番で作業するかを事前に定義
  • ロールバック計画: 万が一リリースに失敗した場合、元の状態に戻す手順
リリース計画書 サンプル
リリース日時: 2026年1月15日（木）23:00〜翌1:00
担当: インフラ担当 山田、バックエンド担当 田中

■ 作業手順
  23:00 メンテナンス画面に切り替え
  23:05 DBバックアップ取得
  23:20 バックアップ確認 → OK
  23:25 マイグレーション実行（DBスキーマ変更）
  23:40 アプリケーションデプロイ

  01:00 監視継続（1時間）

■ ロールバック基準
  ・動作確認でA判定のバグが発見された場合
  ・ロールバック実施者: 山田

■ ロールバック手順
  1. アプリケーションを旧バージョンに戻す
  2. DBマイグレーションをロールバック
  3. バックアップから復元（必要な場合）
  4. 動作確認

リリースの種類と戦略

  • ビッグバンリリース: 全機能を一度にリリース（ウォーターフォールに多い）
  • 段階的リリース: 一部のユーザーから徐々に展開
  • ブルーグリーンデプロイ: 本番環境を2つ用意し切り替える（ダウンタイムなし）
補足: 本番リリースは失敗が許されない場面。不安な点は事前に確認し、作業手順を必ずリハーサルする。

保守とは

リリース後のシステムを安定して動かし続けること。バグ修正・機能追加・性能改善などが含まれる。

種類 内容
障害対応 本番環境のバグ・障害を緊急修正する
機能追加 新しい要望に応じて機能を拡張する
性能改善 遅くなった処理を最適化する
セキュリティ対応 脆弱性のパッチ適用・ライブラリ更新

  2. アプリケーションを再起動

  ・ディスク使用率80%でアラートを設定する
  ・不要ログの自動削除バッチを導入する
  ・ログ保存期間を見直す（30日→7日）

ウォーターフォールを振り返る・次のステップ

ウォーターフォールはプロセスが明確で大規模開発に向く一方、変化に弱い面がある。

観点 ウォーターフォール アジャイル
進め方 工程を順番に 短いサイクルで繰り返す
要件変更 対応しにくい 柔軟に対応できる
向いている場面 要件が明確な大規模案件 変化が多いWebサービス等
ドキュメント 重視 動くソフトウェアを重視
ポイント: どちらが優れているということはない。案件の性質・チームの規模・顧客の特性によって選ぶ。現場ではウォーターフォールとアジャイルを組み合わせた「ハイブリッド型」も多い。

確認問題

問1 Easy

以下の文の（　　　）に当てはまる言葉を答えてください。

「リリース時に失敗した場合に元の状態に戻すことを（　　　　　　）という。事前にその手順を定義した計画を（　　　　　　）という。」
答えを見る
正解: ロールバック / ロールバック計画

ロールバックとは「以前の安定した状態に巻き戻すこと」。本番リリースは「いざとなれば戻せる」状態にしてから実施することが鉄則。

特にDBのスキーマ変更（マイグレーション）を伴う場合は、ロールバック手順が複雑になるため、事前に必ず手順を確認しておく。

問2 Easy

障害対応の流れとして正しい順番はどれか？
  • a) 原因調査 → 障害検知 → 修正 → 一次対応
  • b) 障害検知 → 一次対応 → 原因調査 → 修正 → 再発防止策
  • c) 障害検知 → 原因調査 → 一次対応 → 修正
答えを見る
正解: b)

障害検知 → 一次対応 → 原因調査 → 修正・デプロイ → 再発防止策 → 報告書作成

一次対応（トリアージ）は「まず影響範囲を最小化する」ことが目的。原因が分からなくてもメンテナンスモードにしたり、旧バージョンに戻したりして被害を抑える。原因調査はその後に行う。

問3 Medium

以下の作業を「障害対応・機能追加・性能改善・セキュリティ対応」に分類してください。

  • a) ライブラリの脆弱性パッチを適用する
  • b) ユーザーから要望のあった検索フィルター機能を追加する
  • c) 本番DBに接続できないというアラートが発生し対応する
  • d) 検索に5秒かかっているのを0.5秒に改善する
答えを見る
正解:

・a) → セキュリティ対応
  脆弱性パッチの適用は放置するとシステムが攻撃を受けるリスクがある。定期的な実施が必要。

・b) → 機能追加
  新しい機能を追加する開発作業。要望・仕様確認・設計・実装・テストのサイクルで進める。

・c) → 障害対応
  本番環境で発生した問題への緊急対応。スピードと正確さが求められる。

・d) → 性能改善
  既存機能の速度・効率を向上させる作業。クエリ最適化・キャッシュ導入などが手段となる。

問4 Medium

リリース手順を計画書にまとめずに、当日その場で対応しようとすることの問題点を3つ挙げてください。
答え例を見る
答え例（3つ以上）:

1. 手順漏れのリスク
  バックアップ取得やマイグレーション実行など、重要な手順を当日の緊張やプレッシャーで抜かしてしまう可能性がある。

2. 担当者間の認識齟齬
  「次は誰が何をするのか」が明確でないため、二重作業や抜け漏れが発生しやすい。

3. ロールバック判断が遅れる
  「いつロールバックすべきか」の基準を決めていないと、問題が発生しても判断に時間がかかり被害が拡大する。

4. ミスの記録が残らない
  計画書がないと、実際に何をしたかの記録が残らず、後から振り返りや再発防止ができない。

問5 Hard

考えてみよう

本番リリース直後に重大なバグが発見されました。

「ロールバックするべきか、バグを修正してから再リリースするべきか」を判断するには、どんな情報が必要ですか？
答え例を見る
判断に必要な情報（答え例）:

バグ側の情報
・バグの影響範囲（全ユーザーか、一部の操作のみか）
・バグのランク（A:重大 / B:中 / C:軽）
・修正の難易度（すぐ直せるか、調査に時間がかかるか）
・修正〜再デプロイまでにかかる時間の見込み

ロールバック側の情報
・DBのスキーマ変更（マイグレーション）を含んでいるか
  → スキーマ変更がある場合、ロールバック自体がリスクを伴う
・ロールバックにかかる時間と手順

ビジネス側の情報
・ユーザーへの影響度合い（業務が止まっているか）
・このまま動かし続けることのリスク（データ破損・セキュリティ）
・経営判断が必要かどうか（サービス停止の判断など）

判断の基本原則:
ユーザーへの被害が大きく・修正に時間がかかる場合 → ロールバック優先
軽微なバグで・すぐに修正できる場合 → 修正・再デプロイを検討
