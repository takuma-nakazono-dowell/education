基本設計（外部設計）とは

要件定義で決めた「何を作るか」を受け取り、「どう見せるか・どう構成するか」を決める工程です。 ユーザーから見える部分（外から見える部分）の設計をするため、「外部設計」とも呼ばれます。

ここで作るドキュメントは、顧客と開発チームの両方が承認する重要な成果物です。 「こんな画面で・こんなDBで・こんなAPIで作ります」という合意書にあたります。

基本設計のアウトプット

  • 画面設計書（UI設計）: 各画面のレイアウト・項目・遷移先・バリデーション
  • 画面遷移図: 画面の繋がりを図で表す（ユーザーがどう画面を移動するか）
  • DB設計書（論理設計）: テーブル構成・カラム・主キー・外部キー・関係
  • API設計書: 外部システムや画面とサーバー間のインターフェース定義
  • システム構成図: サーバー・ネットワーク・外部サービスの構成図

画面設計書（ワイヤーフレーム）

実際の見た目（色・フォント）ではなく、「何がどこに配置されるか」を示す設計図です。 モックアップやワイヤーフレームとも呼ばれます。 早い段階で顧客に確認してもらい、認識のズレを修正することが目的です。

DB設計（論理設計・ER図）

データをどのようなテーブルに分けて管理するかを設計します。 「誰が・何を・いつ」持つのかを整理し、テーブル同士の関係を定義します。

ER図の例（簡略）

ユーザー
user_id (PK)
name
email
password
—1:N—
注文
order_id (PK)
user_id (FK)
item_id (FK)
quantity
order_date
—N:1—
商品
item_id (PK)
name
price
stock

PK = 主キー（そのテーブルの行を一意に識別するキー）
FK = 外部キー（他のテーブルの主キーを参照するキー）

論理設計と物理設計の違い
論理設計（基本設計）では「何のデータを持つか・どう関係するか」を決めます。 物理設計（詳細設計）では「どのDBMSに・どんな型で・どんなインデックスで」を決めます。 基本設計ではまず論理設計を固めることに集中します。

正規化とは

「同じデータを複数の場所に持たない設計」のことです。データの重複と矛盾を防ぐための設計原則です。

悪い例: 正規化されていないテーブル
注文テーブルに「商品名・商品価格」を直接書くと、商品の名前が変わった時に 全ての注文行を更新しなければなりません。→ 商品テーブルを別に作り、商品IDで参照するべき。

API設計書

システム間や画面とサーバー間のインターフェースを定義します。 「このURLに、この形式でリクエストを送ると、この形式でレスポンスが返ってくる」を明文化します。

API設計書 サンプル
エンドポイント: GET /api/products
概要: 商品一覧を取得する
権限: ログイン済みユーザー

リクエスト:
  クエリパラメータ:
    keyword (string, 任意): 検索キーワード
    page (integer, 任意): ページ番号（デフォルト: 1）

レスポンス（200 OK）:
  {
    "total": 50,
    "items": [
      { "id": 1, "name": "商品A", "price": 1000 },
      { "id": 2, "name": "商品B", "price": 2500 }
    ]
  }

エラー（401 Unauthorized）:
  { "error": "認証が必要です" }

エラー（500 Internal Server Error）:
  { "error": "サーバーエラーが発生しました" }
基本設計書は「合意書」である
基本設計書は、顧客と開発チームの両方が承認する重要なドキュメントです。 ここで認識のズレを解消しておかないと、後から「こんな画面じゃない」「このDBでは機能が実現できない」 などの大きなやり直しが発生します。承認後の変更は変更管理のプロセスを踏む必要があります。

確認問題

問1 易しい

基本設計のアウトプットを選べ（複数選択）

基本設計（外部設計）で作成するドキュメントとして正しいものをすべて選んでください。

a) 画面設計書
b) テスト結果報告書
c) DB設計書（論理設計）
d) クラス図
e) 画面遷移図
答えを見る
答え: a, c, e

a) 画面設計書 → 基本設計のアウトプット
b) テスト結果報告書 → テスト工程のアウトプット（基本設計ではない）
c) DB設計書（論理設計）→ 基本設計のアウトプット
d) クラス図 → 詳細設計のアウトプット（基本設計ではない）
e) 画面遷移図 → 基本設計のアウトプット

覚え方: 基本設計は「ユーザーから見える部分」。 画面・遷移・データの構成が主なアウトプットです。
問2 易しい

穴埋め問題

ER図において、他のテーブルの主キーを参照するキーのことを（　　）という。 また、そのテーブル自身の行を一意に識別するキーを（　　）という。
答えを見る
答え: 外部キー（FK）・主キー（PK）

・主キー（PK: Primary Key）: そのテーブルの各行を一意に識別するキー
・外部キー（FK: Foreign Key）: 別のテーブルの主キーを参照するキー

例: 注文テーブルの「user_id」は、ユーザーテーブルの「user_id(PK)」を参照する外部キーです。 これにより「この注文は誰の注文か」を知ることができます。
問3 普通

設計の問題点を指摘せよ

以下のテーブル設計の問題点を答えてください。どう改善すべきかも考えてみましょう。

注文テーブル
order_id | user_name | user_email | product_name | product_price | quantity
答えを見る
問題点: ユーザー情報・商品情報が正規化されていない

1. ユーザーが名前やメールアドレスを変更した場合、全ての注文行を更新しなければならない
2. 商品名・価格が変わった場合も同様に全行更新が必要
3. 同じユーザー情報・商品情報が注文の数だけ重複して保存される（データの無駄）
4. 更新を忘れると矛盾したデータが生まれる（データの不整合）

改善案:
・ユーザーテーブル（user_id, name, email）を別に作る
・商品テーブル（product_id, name, price）を別に作る
・注文テーブルには user_id(FK)・product_id(FK)・quantity を持たせる

こうすることで、ユーザー情報・商品情報の変更は各テーブルの1行だけ変えればよくなります。

問5 難しい

APIを設計してみよう

「ユーザーを新規登録するAPI」の設計書を書いてみましょう。

以下を含めること：
・エンドポイントとHTTPメソッド
・リクエストのパラメータ（送るデータ）
・成功時のレスポンス
・エラーケース（最低2つ）
答えのサンプルを見る 
エンドポイント: POST /api/users
概要: 新規ユーザーを登録する
権限: 認証不要（誰でもアクセス可）

リクエスト（リクエストボディ・JSON形式）:
  {
    "name": "田中 太郎",          // 必須・文字列
    "email": "tanaka@example.com", // 必須・メール形式
    "password": "password123"      // 必須・8文字以上
  }

レスポンス（201 Created）:
  {
    "user_id": 42,
    "name": "田中 太郎",
    "email": "tanaka@example.com",
    "created_at": "2025-04-17T10:00:00Z"
  }

エラー（400 Bad Request）:
  入力値が不正な場合
  { "error": "メールアドレスの形式が正しくありません" }

エラー（409 Conflict）:
  メールアドレスが既に登録済みの場合
  { "error": "このメールアドレスはすでに使用されています" }

エラー（500 Internal Server Error）:
  サーバー側の問題
  { "error": "サーバーエラーが発生しました" }
設計のポイント：
・新規作成には POST メソッドを使う（GET は取得、PUT/PATCH は更新、DELETE は削除）
・成功時は 201（Created）を返す（200 でもよいが 201 がより正確）
・エラーの種類に応じて適切なHTTPステータスコードを使い分ける
