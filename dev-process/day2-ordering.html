開発が始まる前の流れ

システム開発の現場では、コードを書く前に必ず「受注プロセス」があります。 顧客がシステムを必要とし、それを作れる開発会社（ベンダー）を選び、契約してからプロジェクトが始まります。

RFP（提案依頼書）とは

RFPとは「Request for Proposal（提案依頼書）」の略です。 顧客（発注者）が、システムを作ってほしい内容をまとめてベンダーに伝える文書です。 「こんなシステムが欲しい。見積もりと提案を出してください」という依頼書です。

複数のベンダーにRFPを送り、提案内容・費用・実績などを比較して1社を選ぶのが一般的です（コンペ方式）。

RFP サンプル
件名: 顧客管理システム新規開発 提案依頼書
発行日: 2025年4月1日
発注者: 株式会社○○商事

■ 背景と目的
現在エクセルで管理している顧客情報を、Webシステムで一元管理したい。
営業担当者がどこからでもアクセスできる環境を整備する。

■ 必要な機能（概要）
・顧客情報の登録・検索・更新・削除
・商談履歴の管理
・月次レポートの自動生成

■ スケジュール要件
・提案書提出期限: 2025年5月15日
・システム稼働希望: 2026年1月

■ 予算感
1,000万円〜1,500万円（目安）

見積もりとは

ベンダーが、要求された機能を作るのにかかる費用と期間を計算して提示する文書です。 RFPに対する「いくらで・いつまでに・何人で作ります」という回答です。

見積もりの考え方

用語 意味
工数（人日・人月） 1人が1日かかる作業量＝1人日。10人日＝「1人が10日」または「2人が5日」
工程別見積もり 設計・製造・テストそれぞれ何人日かかるかを工程ごとに計算する
リスク係数 不確実性に備えて余裕分（バッファ）を加える。一般的に10〜20%
概算見積書 サンプル
件名: 顧客管理システム新規開発
見積日: 2025年4月30日
有効期限: 2025年5月30日

■ 費用内訳
┌────────────┬──────┬─────────┐
│ 工程       │ 工数  │ 金額    │
├────────────┼──────┼─────────┤
│ 要件定義   │ 20人日│  100万円│
│ 基本設計   │ 30人日│  150万円│
│ 詳細設計   │ 40人日│  200万円│
│ 製造       │ 80人日│  400万円│
│ テスト     │ 40人日│  200万円│
│ 合計       │210人日│1,050万円│
└────────────┴──────┴─────────┘

■ 前提条件
・要件が大幅に変更された場合は追加見積もりを行う
・インフラ費用は別途
・本見積もりはRFP記載の要件を前提とする

契約の種類

請負契約

  • 成果物を納品する義務がある
  • バグは直す義務がある（瑕疵担保責任）
  • 金額は固定が多い
  • 「動くシステムを渡すまでが仕事」

準委任契約（SES）

  • 作業時間に対して報酬が発生
  • 成果物の保証はなし
  • 金額は変動（時間×単価）
  • 「作業をすることが仕事」
なぜ仕様変更が追加費用になるのか
新人のうちは契約形態を気にする機会は少ないですが、「なぜ仕様変更は追加費用になるのか」 の理由を知るには請負契約の概念が重要です。 請負契約では「RFPや設計書に書かれた内容を作る」という約束で金額が決まっています。 後から追加の機能を依頼するのは「追加の約束」なので、別の費用が発生します。

キックオフミーティング

契約後、プロジェクトの開始を宣言する会議がキックオフミーティングです。 顧客・開発チーム全員が集まり、プロジェクトの全体像を共有します。

キックオフで確認すること

  • プロジェクトの目的・ゴール（なぜ作るのか、何を達成するのか）
  • 体制図（誰がPMか、どのチームが何を担当するか）
  • スケジュール（WBS・マイルストーン・各工程の開始・終了日）
  • コミュニケーションのルール（報告頻度・連絡手段・会議体）
  • リスクと対応策（想定されるリスクと、それが起きた時の対応方針）

なぜ見積もりは難しいのか

ソフトウェアは形がないため、完成品のイメージがブレやすいです。 「ログイン機能を作る」と一口に言っても、 ・2段階認証があるかどうか ・SNSログインに対応するかどうか ・パスワードリセット機能の複雑さ などで工数が大きく変わります。

要件が曖昧なまま見積もると後から追加費用が発生しやすくなります。 だから、要件定義は見積もりより前か、並行して精度を上げながら進めるのが現実です。

注意: 見積もりトラブルを防ぐには
「とりあえず安く見積もって受注し、後から追加請求」はトラブルの元です。 誠実な見積もりと、変更時のルール（変更管理）を最初に決めておくことが重要です。 変更が発生したときの手続き（変更依頼書の提出・承認フロー）をキックオフ時に確認しておきましょう。

確認問題

問1 易しい

RFPを出すのは誰か

RFP（提案依頼書）を発行するのは誰か？
  • a) ベンダー（システム開発会社）
  • b) 顧客（発注者）
  • c) プロジェクトマネージャー
  • d) プログラマー
答えを見る
答え: b（顧客・発注者）

RFPは「提案してほしい」という依頼書なので、依頼する側＝顧客（発注者）が出します。

ベンダー（開発会社）はRFPを受け取って提案書・見積もりを作成します。 これを複数社に送って比較するのが一般的です。
問2 易しい

穴埋め問題: 工数の計算

1人が5日かかる作業を2人でやると（　　）人日の工数になる。 また、かかる日数は（　　）日になる。
答えを見る
答え: 10人日・2.5日（理想値）

工数（人日）は「人数 × 日数」で計算します。
1人 × 5日 ＝ 5人日
2人でやる場合も工数は変わらず 5人日ですが…

実際には「1人が5日かかる作業」を2人でやれば約2.5日で終わります。
ただし、これは作業が完全に並列できる場合の理想値です。
コミュニケーションコストや依存関係があると、単純に半分にはなりません。

工数（人日）= 1人が5日 = 5人日
2人で並行作業した場合の期間 = 2.5日（理想）
問3 普通

契約の違いを選べ

請負契約と準委任契約の違いとして正しいのはどれか？
  • a) 請負は成果物の納品義務があるが、準委任はない
  • b) 準委任は予算が固定だが、請負は変動する
  • c) 請負はバグを直す義務がないが、準委任はある
  • d) どちらも全く同じである
答えを見る
答え: a

請負契約は「成果物（動くシステム）を納品する」義務があり、バグも直す義務があります。
準委任契約は「作業をする」契約なので、成果物の保証はありません。

b) は逆です。請負の方が固定金額になりやすく、準委任は時間×単価で変動します。
c) は逆です。バグを直す（瑕疵担保責任）のは請負契約です。
問4 普通

考えてみよう: 仕様変更への対応

顧客から「機能を追加したい」という要望が製造工程中に届きました。 どう対応すべきか、また何を確認すべきかを考えてみましょう。
答え例を見る
対応の流れ（例）：

1. まず記録する
「口頭でOK」は禁物。変更依頼を文書（変更依頼書）にまとめます。

2. 影響範囲を確認する
・その変更はどの画面・DBテーブル・処理に影響するか
・既に作成済みの設計書や製造済みのコードを修正する必要があるか

3. 追加費用と期間を見積もる
・変更による追加工数はどのくらいか
・スケジュールにどう影響するか

4. 顧客に承認を取る
・追加費用と延期について顧客が了承してから実施する
・「口頭でOKもらった」は後からトラブルになりやすい

ポイント: 変更を拒否するのではなく、「影響を正しく伝えて顧客に判断してもらう」姿勢が重要です。
問5 難しい

シナリオ問題: 見積もりの差

あなたは見積もりを担当することになりました。ある機能について、 メンバーAは「3人日」、メンバーBは「10人日」と見積もりました。
この差が生まれる原因として考えられることを3つ挙げてください。 また、どう対処しますか？
答え例を見る
差が生まれる原因（例）：

1. 要件の理解の差
Aは「シンプルな実装」を想定し、Bは「エラー処理・例外対応・ログ記録なども含む」 と解釈している。

2. 技術スキルの差
Aは経験豊富で素早く実装できるが、Bは不慣れな技術スタックなので時間がかかると想定している。

3. リスク認識の差
Bは「設計変更が入るかも・テストで問題が出るかも」という不確実性を見積もりに含めている。

対処法：
・見積もりの根拠を共有する（「何を・どうやって作ると思っているか」を確認）
・WBSに落として各作業に分解して再見積もりする
・前提条件（どこまで実装するか・何を含むか）を明確にしてから改めて見積もる

教訓: 見積もりの数字だけを合わせても意味がありません。 「何を・どうやって作るか」の認識を揃えることが重要です。
