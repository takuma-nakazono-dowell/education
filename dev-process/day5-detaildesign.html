詳細設計（内部設計）とは

基本設計で決めた「どう見せるか」をさらに深掘りし、プログラマーが実装できるレベルまで落とし込む工程。ユーザーには見えない内部の動き（処理ロジック・クラス構造・DB物理設計）を定義する。

詳細設計のアウトプット

  • 詳細設計書: 各機能の処理手順・条件分岐・エラー処理
  • クラス図: クラスの構造と関係
  • シーケンス図: 処理の流れ（時系列）
  • DB物理設計書: テーブル定義書（型・制約・インデックス）

クラス図

システムを構成するクラス（設計の単位）とその関係を図で表す。

クラス図（簡略テキスト表現）
┌──────────────────┐    ┌──────────────────┐
│   UserService    │    │  UserRepository  │
│──────────────────│    │──────────────────│
│ + login()        │───>│ + findByEmail()  │
│ + logout()       │    │ + save()         │
│ + register()     │    │ + delete()       │
└──────────────────┘    └──────────────────┘
         │
         ▼
┌──────────────────┐
│      User        │
│──────────────────│
│ - id: int        │
│ - name: string   │
│ - email: string  │
│ - password: hash │
│──────────────────│
│ + validate()     │
└──────────────────┘
ポイント: クラス図は「設計の地図」。複数人で開発する際、誰が何を作るかの分担と依存関係を明確にする。

シーケンス図

処理がどの順番で、どのコンポーネント間でやりとりされるかを時系列で示す図。

シーケンス図：ログイン処理
ユーザー     画面      サーバー      DB
  │          │          │            │
  │─ ID/PW入力→│          │            │
  │          │─ POST /login→          │
  │          │          │─ findByEmail()→│
  │          │          │←─ User データ─│
  │          │          │            │
  │          │          │（パスワード照合）
  │          │          │            │
  │          │←─ 200 OK（トークン）─  │
  │←─ ホーム画面表示─│  │            │
  │          │          │            │

  ※ 失敗時: ←─ 401 エラー → エラーメッセージ表示

DB物理設計書（テーブル定義書）

論理設計（ER図）を具体的なDBMSに落とし込んだ定義書。

テーブル定義書
テーブル名: users
説明: 会員情報を管理するテーブル

┌────────────┬──────────────┬──────────┬────────┬────────────────────────┐
│ カラム名   │ データ型      │ NULL     │ デフォ │ 備考                   │
├────────────┼──────────────┼──────────┼────────┼────────────────────────┤
│ id         │ INT          │ NOT NULL │ AUTO   │ 主キー・自動連番        │
│ name       │ VARCHAR(100) │ NOT NULL │ ─      │ 氏名                   │
│ email      │ VARCHAR(255) │ NOT NULL │ ─      │ UNIQUE制約・ログインID  │
│ password   │ VARCHAR(255) │ NOT NULL │ ─      │ bcryptハッシュ          │
│ created_at │ DATETIME     │ NOT NULL │ NOW()  │ 登録日時               │
│ deleted_at │ DATETIME     │ NULL     │ ─      │ 論理削除用              │
└────────────┴──────────────┴──────────┴────────┴────────────────────────┘
インデックス: email（UNIQUE）

処理仕様書

個々の機能がどんな条件で何を行うかを文章・フローチャートで記述する。

処理仕様書
機能名: ログイン処理
機能ID: FNC-001

■ 入力
  - メールアドレス（文字列）
  - パスワード（文字列）

■ 処理フロー
  1. 入力値の空チェック
     → どちらかが空の場合: エラーコード E001 を返す
  2. メールアドレスでユーザーを検索
     → 存在しない場合: エラーコード E002 を返す
  3. パスワードをハッシュ化して照合
     → 不一致の場合: 失敗回数をカウントし E002 を返す
     → 5回失敗した場合: アカウントをロックし E003 を返す
  4. セッショントークンを発行して返す

■ エラーコード定義
  E001: 入力値不正
  E002: 認証失敗
  E003: アカウントロック
補足: 詳細設計書が丁寧であるほど、プログラマーはロジックではなく「コードに落とす作業」に集中できる。設計書の品質がコードの品質に直結する。
注意: 詳細設計書があっても、製造工程で仕様の解釈がブレることはある。不明点はすぐ設計者に確認するのが新人の重要なスキル。

確認問題

問1 Easy

詳細設計で作成するドキュメントとして正しいものを全て選べ。
  • a) 画面遷移図
  • b) テーブル定義書（DB物理設計）
  • c) シーケンス図
  • d) RFP（提案依頼書）
  • e) 処理仕様書
答えを見る
正解: b, c, e

・b) テーブル定義書: DB物理設計書として詳細設計の主要アウトプット。
・c) シーケンス図: 処理の時系列フローを示す詳細設計の成果物。
・e) 処理仕様書: 各機能の条件分岐・エラー処理を記述する。

a) 画面遷移図は基本設計（外部設計）のアウトプット。
d) RFPは発注前の要件提示文書で、開発工程の設計書ではない。

問2 Easy

以下の文の（　　）に当てはまる言葉を答えてください。

「シーケンス図は、処理がどの順番で、どのコンポーネント間でやりとりされるかを（　　　）で表した図である。」
答えを見る
正解: 時系列

シーケンス図の特徴は「時間の流れ」に沿って処理を追えること。縦軸が時間軸になっており、どのコンポーネントがいつ何をするかを一目で把握できる。

問3 Medium

テーブル定義書の以下のカラム定義を見て答えてください。

deleted_at | DATETIME | NULL | ─ | 論理削除用

Q: deleted_at カラムが NULL許容で設定されている理由は何か？
答えを見る
正解: 論理削除のため

データを物理的に削除（DELETE文でレコードを消す）せず、deleted_at に削除日時を入れることで「削除済み」を表現する手法。

NULLのとき = 有効なデータ
値が入っているとき = 削除済みのデータ

メリット:
・削除したデータを復元できる
・削除の履歴が残る
・関連テーブルへの影響を最小限にできる

問4 Medium

ログインのシーケンス図と処理仕様書を参照して答えてください。

Q: パスワードが5回間違えた場合の処理の流れを、登場するコンポーネントを含めて説明してください。
答えを見る
正解例:

1. ユーザーが誤ったパスワードを入力し、画面を通じてサーバーにPOSTする。
2. サーバーはDBからユーザーデータを取得し、パスワードを照合。不一致のたびに失敗回数をカウントする。
3. 5回目の失敗時点でサーバーがアカウントをロック状態に変更（DBを更新）。
4. サーバーはエラーコード E003（アカウントロック） を画面に返す。
5. 画面はユーザーにアカウントロックのエラーメッセージを表示する。

ロック後は正しいパスワードでもログインできない点がポイント。

問5 Hard

処理仕様書を書いてみよう

「パスワードリセット機能」の処理仕様書を、以下の項目で書いてみましょう。
・入力（何を受け取るか）
・処理フロー（ステップごとに）
・エラーケース（どんな場合にエラーを返すか）
答えのサンプルを見る
答えのサンプル:

■ 入力
・登録済みメールアドレス（文字列）

■ 処理フロー
1. 入力値の空チェック → 空の場合: E001 を返す
2. メールアドレスの形式チェック → 不正形式: E001 を返す
3. メールアドレスでユーザーを検索 → 存在しない場合: E002 を返す
4. リセット用トークン（有効期限付き）を発行してDBに保存
5. リセットURLをメールで送信
6. ユーザーがリセットURLにアクセス → トークンの有効期限チェック
   → 期限切れ: E003 を返す
7. 新しいパスワードを入力・確認入力と一致チェック
8. パスワードをハッシュ化してDBに保存・トークンを無効化

■ エラーコード定義
E001: 入力値不正
E002: メールアドレス未登録
E003: トークン有効期限切れ

ポイント: リセットURLは使い切り（1回使ったら無効）にするのがセキュリティ上の基本。
