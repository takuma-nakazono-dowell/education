要件定義とは

要件定義とは、システムに「何をさせるか」を決める工程です。 顧客と一緒に行い、開発チーム全員の共通認識となる文書（要件定義書）を作ります。 ウォーターフォール開発において、最も重要な工程のひとつです。

重要: 要件定義の品質が全体の品質を決める
要件定義の品質が開発全体の品質を決めます。ここでのミスや曖昧さは、 後工程になるほど大きなコストとなって返ってきます。 「後で決めればいい」という曖昧な要件は、後から設計変更・実装やり直しの原因になります。

機能要件と非機能要件

機能要件: システムが「何をするか」

ユーザーがシステムに対して期待する具体的な機能・操作のことです。

  • ユーザーはメールアドレスとパスワードでログインできる
  • 管理者は商品を登録・編集・削除できる
  • 月次売上レポートをPDF出力できる
  • ユーザーはパスワードをリセットするメールを受け取れる

非機能要件: システムが「どうあるべきか」

機能ではなく、システム全体の品質・特性に関する要件です。

種類 例
性能 検索結果を3秒以内に返す
可用性 稼働率99.9%以上（年間8時間以内の停止）
セキュリティ パスワードはハッシュ化して保存する
保守性 障害発生から2時間以内に復旧できる
スケーラビリティ 同時接続1,000人に対応する
注意: 非機能要件を数値で定義せよ
非機能要件を曖昧にすると、リリース直前に「遅すぎる」「落ちやすい」などの問題が発覚します。 「高速に動作すること」ではなく「検索結果を3秒以内に返す」のように、 必ず数値で定義することが重要です。

ユースケース

「誰が（アクター）、何のために（目的）、何をするか（操作）」を整理したものです。 機能を箇条書きにするだけでなく、使われる文脈・流れを記述することで、 機能の漏れや矛盾を発見しやすくなります。

ユースケース記述 サンプル
ユースケース名: 商品を検索する
アクター: 一般ユーザー
事前条件: ユーザーがログイン済みである

主シナリオ:
  1. ユーザーが検索キーワードを入力する
  2. システムが該当商品の一覧を表示する
  3. ユーザーが商品を選択する
  4. システムが商品詳細画面を表示する

代替シナリオ:
  2a. 該当商品が0件の場合
      → 「該当する商品はありません」を表示する
  2b. システムエラーが発生した場合
      → エラーメッセージを表示し、再試行を促す

事後条件: 商品詳細が表示されている

要件定義書に含まれるもの

  • システム概要: 目的・背景・対象ユーザー・スコープ（何を作り、何を作らないか）
  • 業務フロー図: 現状の業務フロー（As-Is）と理想の業務フロー（To-Be）
  • 機能一覧: 必要な機能の一覧と優先度（Must / Should / Nice-to-have）
  • 画面一覧: どんな画面があるか（詳細レイアウトは基本設計で）
  • データ項目: 扱う情報の種類と概要（詳細なDB設計は基本設計で）
  • 非機能要件: 性能・セキュリティ・可用性・保守性など（数値で）
  • 制約条件: 使用する技術・連携するシステム・法律的制約

受入基準（アクセプタンスクライテリア）

「この条件を満たせば完成と認める」という基準を最初に決めておくものです。 テスト工程の終わりに、顧客がこの基準を使って動作確認（受入テスト）を行います。

受入基準 サンプル

機能: ログイン機能

  • 正しいメールアドレスとパスワードでログインできる
  • 誤ったパスワードを5回入力するとアカウントがロックされる
  • ログアウトすると全てのセッションが無効になる
  • パスワードはデータベースに平文で保存されない
  • ログインページはHTTPSで通信される
受入基準のメリット
受入基準が最初から決まっていると、テスト設計が楽になり、 品質の認識のズレも防げます。「動いた」「動いてない」の水掛け論がなくなり、 基準に照らして判断できます。また、テスターは受入基準をそのままテストケースに使えます。

確認問題

問1 易しい

機能要件か非機能要件かを分類せよ

次の各要件を「機能要件」か「非機能要件」かに分類してください。

a) ユーザーはパスワードをリセットできる
b) 画面の応答時間は2秒以内であること
c) 管理者は全ユーザーの購入履歴を閲覧できる
d) 個人情報は暗号化して保存すること
答えを見る
答え：
a) 機能要件（ユーザーが「できること」を定義している）
b) 非機能要件（応答時間＝システムの性能に関する要件）
c) 機能要件（管理者が「できること」を定義している）
d) 非機能要件（暗号化＝セキュリティに関する要件）

見分け方のポイント:
「〇〇ができる / 〇〇する」という操作・機能 → 機能要件
「〇〇秒以内 / 〇〇%以上 / 暗号化 / 稼働率」など品質・特性 → 非機能要件
問2 易しい

穴埋め問題

要件定義書に含まれる、「この条件を満たせば完成とみなす」という基準のことを （　　　　）という。英語でも書けたら書いてみよう。
答えを見る
答え: 受入基準（アクセプタンスクライテリア）
英語では「Acceptance Criteria」

プロジェクト終盤の受入テスト（UAT: User Acceptance Test）で、 顧客がこの基準を使って「完成か否か」を判断します。
問3 普通

問題のある要件を指摘せよ

以下の要件の問題点を指摘してください。それぞれどう直すべきかも考えてみよう。

a) 「システムは高速に動作すること」
b) 「ユーザーが使いやすいこと」
c) 「データは安全に保管すること」
答えを見る
問題点: いずれも数値・基準が曖昧

a) 「高速に」が曖昧
→ 「一覧画面の表示は3秒以内、検索結果は5秒以内とする」など数値で定義する

b) 「使いやすい」が曖昧
→ 「初回ログインは5ステップ以内」「新人が3時間以内に使い方を習得できる」 など測定可能な基準にする

c) 「安全に」が曖昧
→ 「パスワードはbcryptでハッシュ化して保存」「通信はTLS1.2以上を使用」 「個人情報は暗号化して保存」など具体的な方式を定義する

ポイント: 「良い / 速い / 安全 / 使いやすい」という言葉は要件になりません。 「どうなれば良いと言えるか」を数値・条件で定義することが重要です。
問4 普通

ユースケース記述を書いてみよう

以下の機能についてユースケース記述を書いてみましょう（簡略版でOK）。

機能: 新規会員登録
名前・メールアドレス・パスワードを入力して登録するシナリオ。 すでに登録済みのメールアドレスの場合のエラーも考えること。
答えのサンプルを見る 
ユースケース名: 新規会員登録
アクター: 未登録ユーザー
事前条件: ユーザーが会員登録ページを開いている

主シナリオ:
  1. ユーザーが名前・メール・パスワードを入力する
  2. ユーザーが「登録する」ボタンをクリックする
  3. システムが入力値を検証する
  4. システムがアカウントを作成する
  5. システムが確認メールを送信する
  6. システムが「登録完了」画面を表示する

代替シナリオ:
  3a. 入力値が不正（空欄・形式エラー）の場合
      → エラーメッセージを表示し、入力フォームを再表示する
  3b. 入力メールがすでに登録済みの場合
      → 「このメールアドレスは使用済みです」を表示する

事後条件: アカウントが作成され、確認メールが送信されている
問5 難しい

シナリオ問題: 曖昧な要求への対応

顧客から「いい感じのダッシュボードを作ってほしい」という要求が届きました。

この要求の問題点と、要件定義でどんな質問をすべきかを考えてください。 最低5つの質問を挙げてみましょう。
答え例を見る
問題点:
「いい感じ」が完全に主観的で、開発者と顧客の想像が全く違う可能性があります。 これをそのまま設計・製造に渡すと、完成品が期待と全く違うものになります。

確認すべき質問の例：

1. 誰が見るダッシュボードですか？（経営者・営業担当・一般ユーザーで設計が変わる）
2. 何のデータを表示したいですか？（売上・在庫・アクセス数・問い合わせ件数など）
3. データはリアルタイム更新が必要ですか？（毎秒 / 1日1回 / 手動更新）
4. どんなグラフや形式で表示したいですか？（棒グラフ・折れ線・数値のみなど）
5. 権限によって見える内容を変えますか？（管理者は全データ、担当者は自分の分のみなど）
6. 参考にしたい既存サービスのスクリーンショットはありますか？
7. 期間の絞り込みは必要ですか？（今日・今月・任意期間）

教訓: 「いい感じ」「使いやすく」「きれいに」という言葉が出たら、 具体的なイメージをヒアリングしましょう。参考サイトや既存システムを見せてもらうのが最も有効です。
