インデックスとは？

データベースの索引です。本の「索引ページ」と同じ仕組みで、膨大なデータの中から目的の行を素早く見つけます。

インデックスなし：テーブルを先頭から1行ずつスキャン（フルスキャン）
インデックスあり：索引を引いて直接その行にジャンプ

100万件のテーブルで、フルスキャン → 数秒、インデックス検索 → 数ミリ秒

インデックスの作成

-- customer_idにインデックスを作成 CREATE INDEX idx_orders_customer ON orders (customer_id); -- 複合インデックス（複数列） CREATE INDEX idx_orders_date_customer ON orders (order_date, customer_id);
インデックスが自動的に作られる列
PRIMARY KEY と UNIQUE 制約の列には自動でインデックスが付きます。

重いクエリの アンチパターン

❌ パターン1: SELECT * で全列取得
SELECT * FROM orders; -- 不要な列まで取得
✅ 改善: 必要な列だけ指定
SELECT id, customer_id, order_date FROM orders;
❌ パターン2: LIKE の前方ワイルドカード
WHERE name LIKE '%田中%' -- インデックス無効化
✅ 改善: 前方一致はインデックスが使える
WHERE name LIKE '田中%'
❌ パターン3: WHERE句で列に関数を適用
WHERE YEAR(order_date) = 2024 -- インデックス無効
✅ 改善: 列をそのまま使う
WHERE order_date >= '2024-01-01' AND order_date < '2025-01-01'
❌ パターン4: インデックスのない列でのJOIN
JOIN order_details ON orders.id = order_details.order_id -- order_id にインデックスがないと遅い
✅ 改善: JOIN列にインデックスを作成
CREATE INDEX idx ON order_details(order_id);
インデックスの注意点
インデックスは検索を速くしますが、INSERT/UPDATE/DELETE は遅くなります。 闇雲に作らず、実際に遅いクエリのWHERE/JOIN列に絞って作成しましょう。
