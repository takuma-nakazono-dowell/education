トランザクションとは？

「一連の処理をまとめて1つの作業として扱う」仕組みです。
途中で失敗したら全部なかったことにできます。

銀行振込の例
① A口座から10,000円引く
② B口座に10,000円足す

①の後にシステムがクラッシュしたら？
→ トランザクションで包めば①も取り消せる

基本構文

BEGIN TRANSACTION; -- 開始 -- 複数のSQL処理 UPDATE accounts SET balance = balance - 10000 WHERE id = 1; UPDATE accounts SET balance = balance + 10000 WHERE id = 2; COMMIT; -- 確定（成功時） -- または ROLLBACK; -- 取り消し（失敗時）

ACID 特性

A — 原子性
全部成功 or 全部失敗。中途半端な状態にならない
C — 一貫性
トランザクション前後でデータの整合性が保たれる
I — 独立性
他のトランザクションの途中状態が見えない
D — 永続性
COMMITしたデータはクラッシュしても消えない

ロックとデッドロック

トランザクション中は、更新対象の行や表がロック（占有）されます。

デッドロックとは
AとBが互いに相手のロックを待ち合って、どちらも進めなくなる状態。
接続A注文テーブルをロック
接続B在庫テーブルをロック
接続A在庫テーブルを待つ... ⏳
接続B注文テーブルを待つ... ⏳ ← 詰み！

長時間トランザクションの危険性

やってはいけないこと
BEGIN したまま長時間放置すると：
・ロックが長時間続き、他の処理が止まる
・デッドロックが発生しやすくなる
・ログが肥大化する（SQL Server等）

ルール：トランザクションは短く終わらせる
ユーザーの操作待ち（画面入力など）の間はトランザクションを開いてはいけない
データベースを準備中...

練習問題（インタラクティブ実験）

実験1: COMMIT で確定

① 「在庫更新 → 確認」をクリックして更新後の在庫を確認します。
② 「COMMIT確定」でデータを確定し、もう一度「確認」で永続化を確認します。

BEGINから順番に実行してみましょう

実験2: ROLLBACK で取り消し

① BEGINして在庫を更新
② ROLLBACKで取り消し
③ 在庫が元に戻っていることを確認

BEGINから順番に実行してみましょう
